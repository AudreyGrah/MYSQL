Modélisation d'une BDD

Avant de créer notre BDD il est essentiel de se poser et de réfléchir à la modélisation. Rien est irréversible, mais changer la modélisation en cours de route est déconseiller car cela forcera certainement à réécrire une partie du code ce qui peut prendre beaucoup de temps et avoir des conséquences sur ce que l'on a déjà fait comme des dérèglements d'autres parties de l'application. C'est la raison pour laquelle il vaut mieux éviter de se tromper

Comment modéliser une BDD

Si la BDD n'est pas complexe, il est possible de la modéliser directement avec une feille et un stylo. CEpendant dans la plupart des cas, nous aurons besoin d'un logiciel pour accompagner et avoir une vue d'ensemble. Un logiciel connu permet de modéliser une BDD, c'est : MySQL Workbench. Il existe cependant de nombreux outils permettant cette manipulation. Pour une bonne conception de sa BDD il faut réfléchir en terme de sujet en modélisant ceux du monde réel. Par exemple, si nous vendons des produits sur notre site, il y aura, une table utilisateurs, une table produit et une table commande (cette table rassemlera l'id de l'utilisateur et celui du produit).

Les tables

  • Un sujet représente une table dans une BDD
  • Une table est un emplacement de sauvegarde
  • Les données sont stockées à l'intérieur de tables. Une table peut être comparée à une liste qui contient des enregistrements relatifs à un sujet bien défini.

Il faut en fait dès le départ, réfléchir à chaque sujet abordé dans notre site et à ses conséquences. Si nous vendons des produits, nous aurons certainement des membres. Une table membre (user) sera donc nécéssaire. En terme de fonctionnalité, si nous souhaitons proposer un abonnement newsletter, nous aurons besoin d'une table newsletter qui sera liée à la table membre (user) 1 table = 1 sujet

Les colonnes ou les champs

Chaque table possède généralement plusieurs champs, aussi appelé colonnes. Les colonnes représentent des caractéristiques relatives au sujet. (-> la table)

Pour savoir quelle colonne mettre dans quelle table, il faut se poser les questions suivantes : qu'est ce qui pourrait décrire mon sujet ? quelles sont les informations sur mon sujet ?
La table user aura par exemple les champs : pseudo, mot de passe, nom, prénom, etc.

Les types de données

Type de champs Description Contexte

TEXTE Chaîne de caractères (illimité) on parle de la valeur Pour un contenu textuel long, comme un article de blog
INT Contenu numérique (chiffre) pour un identifiant, un prix, un code postal, le nmbre de produit en stock etc..

Types de données, PK, AI pour les identifiants

a) Les identifiants, PK (Primary Key - Clé primaire)

  • Les identifiants sont des champs un peu particulier car c'est la seul colonne qui ne décrit pas le sujet. Ils sont cependant systèmatiquement présent dans les chaques tables en première position.
  • Chaque table possèdent une colone identifiant que l'on appelera la clé primaire ou tout simplement "PK"
  • On pourra l'appeler tout simplement "ID" mais on retrouvera parfois "idProduit" ou encore "idMembre", etc.

b) AI, l'auto incrémentation

Auto_Increment est une option permettant de générer un numéro unique dans une colonne de type clé primaire (PK). Par conséquent retenez bien le premier champs de chaque table sera systèmatiquement un "id" qui sera à la fois PK (Primary Key) et AI (Auto_Increment).

c) NULL / NOT NULL

Dans chaque champ, nous pourrons indiquer si nous acceptons les valeurs NULL ou NOT NULL. Null est un type de valeuren informatique permettant de laisser un champ vide si nous n'avons pas d'informations à y laisser

d) Les relations

Ils arrivent parfois que les sujets interagissent entre eux.

  • Exemple : un "membre" commande un "produit", nous devons enregistrer l'informations dans une table "commande"
  • Autre exemple : Un "conducteur" conduit un "véhciule". il faudra une table permettant de préciser quel véhciule va avec quel conducteur

Les cadinalités

Les cadinalités permettent de connaître le chiffre minimum et maximum d'enregistrement pour une relation.

Par exemple, un enfant est la descendance d'une seule mère biologique (minimum = 1 / maximum = 1). En revanche, une femme peut avoir plusieurs enfants ou aucun (minimum = 0 / maximum = n (n = infini) )

Autre exemple, 1 livre possèdent forcément, 1 auteur et 1 max (1,1).

Tables de jointure

Une table de jointure permet de faire le lien entre deux tables. Par exemple, prenons le cas d'une société de taxis qui possèderait des conducteurs (tale : conducteur) et des véhciules (table : véhicule).

  • Un conducteur peut conduire 1 ou plusieurs véhicules
  • Un véhicules peut être conduit par 0 ou plusieurs conducteurs

Comment savoir quelle conducteur conduit quelle véhicules ?

Grâce à la table de jointure, nous savons qu'Alexandre conduit à la fois la Mercedes et la BMW et que Julien ne conduit que la BMW

Les lcés étrangères, Foreign Key (FK)

Les clés étrangères, sont des champs forcément clé primaire dans leur table d'origine et qui sont présent dans une table extérieur.
Un cham clé étrangère (FK) placé à l'extérieur de sa table d'origine permet de faire la relation avec un même sujet

Les enregistrements

Chaque enregistrement représente une ligne dans la BDD et chaque enregistrement est auto incrémenté. Même si un enregistrement est supprimé, son ID ne sera pas réutilisé.

Comment intérroger la BDD ?

a) Les requêtes

Une requête permet de poser une question, d'interroger la BDD ou encore donner un ordre (CRUD -> Create -> Créer / Read -> Lis une information / Update -> met à jour / Delete -> supprimer)

Nous terminerons toutes nos requêtes SQL par un point virgule ";" pour signier qu'il faut exécuter la demande.

b) Types de requêtes

Il existe 4 types de requêtes

  • La sélection : SELECT : c'est une question / réponse => Nous faisons une demande et obtenons une réponse si la demande est bien formulé
  • L'insertion : INSERT : requête d'action -> impact sur les données
  • La modification : UPDATE : requête d'action -> On modifie les données déjà présentes dans notre table.
  • La suppression : DELETE : requête d'action -> on supprime un enregistrement de notre table // une table // une BDD
